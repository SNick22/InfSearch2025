Всем привет, меня зовут Олег, я техлид в ДомКлике. В нашей команде ядром стека является Kotlin и Spring Boot. Хочу поделиться с вами своим опытом по взаимодействию и особенностях работы с PostgreSQL и Hibernate в связке со Spring Boot и Kotlin. Также на примере микросервиса, покажу преимущества Kotlin и его отличия от аналогичного приложения на Java. Расскажу о не совсем очевидных сложностях, с которыми могут столкнуться новички при использовании этого стека с Hibernate. Статья будет полезна разработчикам, желающим перейти на Kotlin и знакомых со Spring Boot, Hibernate Java.<br/>
<a name="habracut"></a><br/>
<h2>Плагины</h2><br/>
Для приложения на Kotlin в качестве сборщика проекта возьмём Gradle Kotlin DSL. Список подключенных плагинов будет стандартным для Spring Boot, а для Kotlin с Hibernate у нас появится несколько новых: <br/>
<br/>
<pre><code class="kotlin">plugins {
    id("org.springframework.boot") version "2.2.7.RELEASE"
    id("io.spring.dependency-management") version "1.0.9.RELEASE"
    kotlin("jvm") version "1.3.72"
    kotlin("plugin.spring") version "1.3.72"
    kotlin("plugin.jpa") version "1.3.72"
}
</code></pre><br/>
Рассмотрим три последних.<br/>
<br/>
<code>kotlin(«jvm»)</code> — базовый плагин Kotlin для работы на JVM. Без которого не заведется ни одно приложение на Java-стеке.<br/>
<br/>
<code>kotlin(«plugin.spring»)</code> — поскольку классы в Kotlin по умолчанию финальны, то этот плагин автоматически сделает классы, помеченные аннотациями <a href="http://docs.spring.io/spring-framework/docs/current/javadoc-api/org/springframework/stereotype/Component.html"><code>@Component</code></a>, <a href="http://docs.spring.io/spring/docs/current/javadoc-api/org/springframework/scheduling/annotation/Async.html"><code>@Async</code></a>, <a href="http://docs.spring.io/spring-framework/docs/current/javadoc-api/org/springframework/transaction/annotation/Transactional.html"><code>@Transactional</code></a>, <a href="http://docs.spring.io/spring-framework/docs/current/javadoc-api/org/springframework/cache/annotation/Cacheable.html"><code>@Cacheable</code></a> и <a href="https://docs.spring.io/spring-boot/docs/current/api/org/springframework/boot/test/context/SpringBootTest.html"><code>@SpringBootTest</code></a> открытыми к наследованию, а в тематике, относящейся этой статье, это позволит классам, написанным на Kotlin быть проксированными в Spring через CGLib прокси.<br/>
<br/>
Важно отметить, что сущности, помеченные аннотациями <code>@Entity</code>, <code>@MappedSuperclass</code> и <code>@Embaddable</code>, не станут <code>open</code> после подключения плагина. Более того, <code>get accessor</code>’ы тоже будут финальными, и тогда <strong>мы потеряем возможность работать</strong> с <code>entity reference</code>. Чтобы этого избежать и сделать <code>Entity</code> и его поля <code>open</code>, добавим в build.gradle.kts:<br/>
<br/>
<pre><code class="kotlin">allOpen {
   annotation("javax.persistence.Entity")
   annotation("javax.persistence.MappedSuperclass")
   annotation("javax.persistence.Embeddable")
}</code></pre><br/>
<code>kotlin(«plugin.jpa»)</code> — Если предыдущие два плагина применяются к любому приложению на Kotlin + Spring Boot, то следующий, уже относится напрямую к Hibernate. А он, как известно, для инициализации <code>Entity</code> использует рефлексию и инициализирует класс с конструктором без аргументов. Но так как мы пишем на Kotlin, такового конструктора может и не найтись. Если мы определили свой собственный первичный конструктор (<i>primary constructor</i>), то при загрузке <code>Entity</code> у нас выкинет исключение:<br/>
<br/>
<pre><code class="plaintext">org.hibernate.InstantiationException: No default constructor for entity
</code></pre><br/>
<h2>Зависимости</h2><br/>
Набор зависимостей у нас тоже будет не совсем идентичный набору на Java:<br/>
<br/>
<pre><code class="kotlin">dependencies {
  implementation("org.springframework.boot:spring-boot-starter-web")
  implementation("org.springframework.boot:spring-boot-starter-data-jpa")
  implementation("com.fasterxml.jackson.module:jackson-module-kotlin")
  implementation("org.jetbrains.kotlin:kotlin-reflect")
  implementation("org.jetbrains.kotlin:kotlin-stdlib-jdk8")
  implementation("org.liquibase:liquibase-core")

  runtimeOnly("org.postgresql:postgresql")

  testImplementation("org.springframework.boot:spring-boot-starter-test")
  testImplementation("org.testcontainers:testcontainers:$testContainersVer")
  testImplementation("org.testcontainers:postgresql:$testContainersVer")
}</code></pre><br/>
Добавим еще пару зависимостей в дополнение к стандартному веб-стартеру Spring Boot и к основному интересующему нас стартеру <code>org.springframework.boot:spring-boot-starter-data-jpa</code>, который в качестве реализации JPA по умолчанию тянет Hibernate: <br/>
<br/>
<code>org.jetbrains.kotlin:kotlin-reflect</code> — нужен для рефлексии на Kotlin, которая уже поддерживается в Spring Boot и широко используется для инициализации классов.<br/>
<br/>
<code>org.jetbrains.kotlin:kotlin-stdlib-jdk8</code> — добавляет возможность работать с коллекциями Java, поддержку стримов и многое другое. <br/>
<br/>
На этом различия в конфигурировании проекта на Kotlin по сравнению с Java у нас заканчиваются, перейдем к самому проекту, его структуре таблиц и сущностей.<br/>
<br/>
<h2>Таблицы и сущности</h2><br/>
Наше приложение будет состоять из двух таблиц department и employee, которые связаны отношением «один ко многим».<br/>
<br/>
Структура таблиц:<br/>
<br/>
<div style="text-align:center;"><img src="https://habrastorage.org/r/w1560/getpro/habr/post_images/53d/260/2a9/53d2602a91d6b89c3fddb16c27bc824a.png" data-src="https://habrastorage.org/getpro/habr/post_images/53d/260/2a9/53d2602a91d6b89c3fddb16c27bc824a.png"/></div><br/>
В качестве базы будем использовать СУБД <i>PostgreSQL</i>. Структуру таблиц создадим с помощью <i>liquibase</i>, а в качестве тестовых зависимостей будем использовать стандартный стартер: <br/>
<br/>
<code>org.springframework.boot:spring-boot-starter-test</code> — тестировать будем в Docker с помощью <i>testcontainers.</i><br/>
<br/>
<h4>Сущности</h4><br/>
Как и в любом приложении с более чем одной сущностью, создадим общего предка для всех <code>entity</code>.<br/>
<br/>
<code>BaseEntity</code>:<br/>
<br/>
<pre><code class="kotlin">@MappedSuperclass
abstract class BaseEntity&lt;T&gt; {

   @Id
   @GeneratedValue(strategy = GenerationType.IDENTITY)
   var id: T? = null

   override fun equals(other: Any?): Boolean {
       other ?: return false

       if (this === other) return true

       if (javaClass != ProxyUtils.getUserClass(other)) return false

       other as BaseEntity&lt;*&gt;

       return this.id != null &amp;&amp; this.id == other.id
   }

   override fun hashCode() = 25

   override fun toString(): String {
       return "${this.javaClass.simpleName}(id=$id)"
   }
}</code></pre><br/>
<code>DepartmentEntity</code>:<br/>
<br/>
<pre><code class="kotlin">@Entity
@Table(name = "department")
class DepartmentEntity(

       val name: String,

       @OneToMany(
               mappedBy = "department",
               fetch = FetchType.LAZY,
               orphanRemoval = true,
               cascade = [CascadeType.ALL]
       )
       val employees: MutableList&lt;EmployeeEntity&gt; = mutableListOf()
) : BaseAuditEntity&lt;Long&gt;() {

   fun addEmployee(block: DepartmentEntity.() -&gt; EmployeeEntity) {
       employees.add(block())
   }

   fun setEmployees(block: DepartmentEntity.() -&gt; MutableSet&lt;EmployeeEntity&gt;) {
       employees.clear()
       employees.addAll(block())
   }
}</code></pre><br/>
<code>EmployeeEntity</code>:<br/>
<br/>
<pre><code class="kotlin">@Entity
@Table(name = "employee")
class EmployeeEntity(

       val firstName: String,

       var lastName: String? = null,

       @ManyToOne
       @JoinColumn(name = "department_id")
       val department: DepartmentEntity
) : BaseAuditEntity&lt;Long&gt;()</code></pre><br/>
<strong>Мы не используем Data-классы</strong>. Это кажется явным преимуществом Kotlin перед Java (до 14 версии), и этому есть объяснение.<br/>
<br/>
<h4>Почему не использовать?</h4><br/>
Data-классы, помимо того, что они финальны сами по себе, имеют по всем полям определенные <code>equals</code>, <code>hashCode</code> и <code>toString</code>. А это <strong>недопустимо</strong> в связке с Hibernate.<br/>
<br/>
Почему? А также зачем <code>hashCode</code> всегда равен константе — ответ в <a href="https://docs.jboss.org/hibernate/orm/5.1/userguide/html_single/chapters/domain/entity.html">документации самого Hibernate</a>. Конкретно нас интересует вот этот раздел: <br/>
<br/>
<blockquote>Although using a natural-id is best for <strong>equals</strong> and <strong>hashCode</strong>, sometimes you only have the entity identifier that provides a unique constraint.<br/>
<br/>
It’s possible to use the entity identifier for equality check, but it needs a workaround:<br/>
<br/>
<ul>
<li>you need to provide a constant value for <strong>hashCode</strong> so that the hash code value does not change before and after the entity is flushed.</li>
<li>you need to compare the entity identifier equality only for non-transient entities.</li>
</ul></blockquote><br/>
То есть сравнивать нужно либо по <code>natural id</code>, либо, как в нашем примере, по <code>primary key id</code>. Это позволит избежать множества проблем при сравнении сущности и убережет от ее потери при использовании сущности в качестве элемента в <code>Set</code>. <br/>
<br/>
Наличие же <code>toString</code>, определенного по всем полям, и вовсе убивает всю ленивость, например, при журналировании сущности, так как будут проинициализированы все поля для вывода в строку.<br/>
<br/>
Учитывая особенности Hibernate, эта функциональность Kotlin нам не подойдет. <br/>
<br/>
<h4>Конструктор класса</h4><br/>
Kotlin позволяет задавать переменным значения через конструктор, чем грех не воспользоваться. Рассмотрим еще раз <code>DepartmentEntity</code>:<br/>
<br/>
<pre><code class="kotlin">class DepartmentEntity(

       val name: String,

       @OneToMany(
               mappedBy = "department",
               fetch = FetchType.LAZY,
               orphanRemoval = true,
               cascade = [CascadeType.ALL]
       )
       val employees: MutableList&lt;EmployeeEntity&gt; = mutableListOf()
) : BaseAuditEntity&lt;Long&gt;() {</code></pre><br/>
Также мы можем проинициализировать через конструктор название подразделения, например:<br/>
<br/>
<pre><code class="kotlin">departmentRepository.save(DepartmentEntity(name = "Department One"))</code></pre><br/>
Через конструктор можно инициализировать, в том числе, и список сотрудников employees. Коллекции, разумеется, объявим изменяемыми. <br/>
<br/>
<h4>Используйте var/val в зависимости от необходимости изменения поля</h4><br/>
Название организации мы пометили как <code>val</code>:<br/>
<br/>
<pre><code class="kotlin">class DepartmentEntity(

       val name: String,</code></pre><br/>
и оно не может быть <code>null</code>. <br/>
<br/>
Выбор <code>var</code>/<code>val</code> является удобной опцией и зависит от бизнес-логики. Выбирать между <code>var</code> и <code>val</code> надо исходя из требования: должно ли поле сущности быть изменяемым.<br/>
<br/>
<h4>Допустимость null в полях только в соответствии с БД</h4><br/>
Насчет допустимости значений <code>null</code> в полях всё не так просто. Ранее мы погрузились немного в глубины Hibernate: говоря о plugin.jpa, я упомянул про использование конструктора без аргументов при инициализации сущности.<br/>
<br/>
При инициализации полей тоже используется рефлексия. И если в базе в соответствующей колонке хранилось значение <code>null</code>, то класс спокойно инициализируется с этим полем со значением <code>null</code>. При обращении к нему мы рискуем получить NPE, хотя поле и помечено как <code>not nullable</code>. Чтобы этого не случилось, надо следить за синхронностью структуры таблиц и классов.<br/>
<br/>
Если посмотреть на описанное в последних двух разделах более комплексно, то эти правила применимы не только к примитивам, но и к связке сущностей. <br/>
<br/>
Например, <code>EmployeeEntity</code> всегда привязан к <code>DepartmentEntity</code>:<br/>
<br/>
<pre><code class="kotlin">class EmployeeEntity(

       val firstName: String,

       var lastName: String? = null,

       @ManyToOne
       @JoinColumn(name = "department_id")
       val department: DepartmentEntity
) : BaseAuditEntity&lt;Long&gt;()</code></pre><br/>
<code>Department</code> является не <code>null</code> и его нельзя изменить, что может избавить от разного рода ошибок, в особенности, если бизнес-логика требует неизменяемости. <br/>
<br/>
<h4>Репозитории</h4><br/>
При использовании Kotlin, у репозиториев из коробки появилась проверка на допустимость <code>null</code>. Так, если мы уверены, что при поиске <code>department</code> по имени результат будет уникальный и единственный, то можно возвращаемый тип указать как <code>non nullable</code>:<br/>
<br/>
<pre><code class="kotlin">interface DepartmentRepository : JpaRepository&lt;DepartmentEntity, Long&gt; {

   fun findOneByName(name: String) : DepartmentEntity
}</code></pre><br/>
Здесь <code>DepartmentEntity</code> указан единственным и не может быть <code>null</code>. Если же по какой-то причине мы не нашли искомый <code>department</code>, то поймаем уже не NPE, а нечто другое:<br/>
<br/>
<pre><code class="plaintext">org.springframework.dao.EmptyResultDataAccessException: Result must not be null!
</code></pre><br/>
Такая обработка достигается с помощью добавления специализированной поддержки Kotlin в <code>MethodInvocationValidator</code> и <code>ReflectionUtils</code> в spring data commons. <br/>
<br/>
<h4>lateinit var</h4><br/>
Ещё одной фичей Kotlin, которую хотелось бы рассмотреть, является <code>lateinit var</code>.<br/>
<br/>
Добавим новый класс-предок: <code>BaseAuditEntity</code>.<br/>
<br/>
<pre><code class="kotlin">@MappedSuperclass
@EntityListeners(AuditingEntityListener::class)
abstract class BaseAuditEntity&lt;T&gt; : BaseEntity&lt;T&gt;() {

   @CreatedDate
   @Column(updatable = false, nullable = false)
   lateinit var created: LocalDateTime

   @LastModifiedDate
   @Column(nullable = false)
   lateinit var modified: LocalDateTime
}</code></pre><br/>
Рассмотрим применение <code>lateinit var</code> на примере полей аудита (<code>created</code>, <code>modified</code>). <br/>
<br/>
<code>lateinit var</code> — это <code>not null</code> поле с отложенной инициализацией. Обращение к полю до его инициализации генерирует ошибку:<br/>
<br/>
<pre><code class="plaintext">kotlin.UninitializedPropertyAccessException: lateinit property has not been initialized
</code></pre><br/>
Как правило, мы обращаемся к полям <code>created</code> и <code>modified</code> уже после того, как сущность была сохранена в БД. В нашем случае, данные в этих поляхпроставляются на этапе сохранения и они <code>not null</code>, то <code>lateinit var</code> нам более чем подходит. <br/>
<br/>
<h2>Итоги</h2><br/>
Мы создали приложение, в котором учтены многие преимущества Kotlin, и рассмотрели важные отличия от Java, избежав многих скрытых сюрпризов. Буду рад, если эта статья окажется полезна не только новичкам. Позднее мы продолжим тему общения микросервиса с БД.<br/>
