xmlns="http://www.w3.org/1999/xhtml"><p><i>Наша команда, аналогично с автором статьи, уже почти как год перешла со Scala на Kotlin в качестве основного языка. Мое мнение во многом совпадает с автором, поэтому предлагаю вам перевод его интересной статьи.</i></p><br/>
<p>Прошло прилично времени с того момента как я не обновлял блог. Вот уже как год я перешел со Scala, моего основного языка, на Kotlin. Язык позаимствовал много хороших вещей, которые мне нравились в Scala, сумев при этом избежать многих подводных камней и неоднозначности, которая есть в Scala. </p><br/>
<p>Ниже я хочу привести примеры, которые мне нравятся в Scala и Kotlin, а также их сравнение в том, как они реализованы в обоих языках. </p><a name="habracut"></a><br/>
<h4><b>Объявление и выведение типов</b></h4><br/>
<p>Что мне особенно нравится в обоих языках так это то, что они оба являются статически типизированными с выведением типов. Это предоставляет вам возможность в полной степени воспользоваться мощью статической типизации без громоздких объявлений в коде (<i>ориг.: declarative boiler plate</i>). В большинстве случаев это работает в обоих языках. В обоих языках также прослеживается предпочтение неизменяемым типам вместе с опциональным объявлением типа переменной после ее названия. </p><br/>
<p>Пример кода будет одинаковый в обоих языках:</p><br/>
<p>Объявление неизменяемой переменной с именем <i>age</i> и типом <i>Int</i>:</p><br/>
<pre><code class="scala">val age = 1 </code></pre><br/>
<p>Объявление изменяемой переменной с типом <i>String</i>:</p><br/>
<pre><code class="scala">var greeting = "Hello"</code></pre><br/>
<p>Оба языка поддерживают лямбда функции как объекты первого класса, которые могут быть присвоены переменным или переданы в качестве параметров функциям:</p><br/>
<p><u>Scala</u></p><br/>
<pre><code class="scala">val double = (i: Int) =&gt; { i * 2 }</code></pre><br/>
<p><u>Kotlin </u></p><br/>
<pre><code class="scala">val double = {i: Int -&gt; i * 2 }</code></pre><br/>
<h4><b><i>Data</i> / <i>Case</i> классы</b></h4><br/>
<p>Scala и Kotlin имеют схожий концепт data классов, которые являются представлением <i>data model object</i>.</p><br/>
<h5><b>Подход в Scala</b></h5><br/>
<p>В Scala это <i>case</i> классы, которые выглядят следующим образом:</p><br/>
<pre><code class="scala">case class Person(name: String, age: Int)</code></pre><br/>
<ul>
<li>Есть <i>apply</i> метод (не нужно использовать ключевое слово <i>new</i> при создание инстанса) <br/>
</li>
<li>Методы для доступа объявлены для каждого <i>property</i> (если <i>property</i> объявлено как <i>var</i> то <i>setter</i> метод также будет присутствовать)<br/>
</li>
<li><i>toString</i>, <i>equal</i> и <i>hashCode</i> разумно объявлены<br/>
</li>
<li>Eсть copy функция<br/>
</li>
<li>Есть <i>unapply</i> метод (который позволяет использовать данные классы в <i>pattern matching</i>)<br/>
</li>
</ul><br/>
<h5><b>Подход в <i>Kotlin</i></b></h5><br/>
<p>Kotlin называет данные классы как <i>data class</i></p><br/>
<pre><code class="scala">data class Person (val name: String, val age: Int)</code></pre><br/>
<p>Ключевые особенности:</p><br/>
<ul>
<li>Методы для доступа объявлены для каждого <i>property</i> (если <i>property</i> объявлено как <i>var</i> то <i>setter</i> метод также будет присутствовать). Это не исключительная особенность <i>data</i> классов, утверждение справедливо для любых классов в Kotlin.<br/>
</li>
<li>Разумно объявлены <i>toString</i>, <i>equal</i> и <i>hashCode</i><br/>
</li>
<li>Сopy функция<br/>
</li>
<li><i>component1..componentN</i> функции. По аналогии используется в качестве <i>unapply</i>.<br/>
</li>
<li>Реализует <i>JavaBean</i> <i>getter</i> и <i>setter</i>, необходимых для таких <i>Java</i> фреймворков как Hibernate, Jackson, без изменений.<br/>
</li>
</ul><br/>
<p>В Kotlin нет необходимости в специальном <i>apply</i> методе, также как и не нужно ключевое слово <i>new</i> для инициализации класса. Так что это стандартное объявление конструктора как и для любых других классов.</p><br/>
<h4><b>Сравнение</b></h4><br/>
<p>В основном <i>case</i> и <i>data</i> классы похожи.</p><br/>
<p>Пример ниже выглядит одинаково в обоих языках:</p><br/>
<pre><code class="scala">val jack = Person("jack", 1)
val olderJack = jack.copy(age = 2)</code></pre><br/>
<p>В целом я нашел <i>data</i> и <i>case</i> классы взаимозаменяемым в повседневном использовании. В Kotlin есть некоторые ограничения на наследование <i>data</i> классов, но это было сделано из благих намерений c учетом реализации <i>equals</i> и <i>componentN</i> функций, чтобы избежать подводных камней. </p><br/>
<p>В Scala <i>case</i> классы более мощные в <i>pattern matсhing</i> по сравнению с тем как Kotlin работает с <i>data</i> классами в ‘when’ блоках, в которых этого не хватает.</p><br/>
<p>Подход Kotlin работает лучше для существующих Java фреймворков, т.к. они выгдядят для них как обычные <i>Java bean</i>.</p><br/>
<p>Оба языка позволяют передавать параметры по имени и позволяют указать значению по умолчанию для них.</p><br/>
<h4><b>Null Safely / Optionality</b></h4><br/>
<h5><b>Подход в Scala</b></h5><br/>
<p>В Scala <i>null safely</i> заключается в использовании монады <i>option</i>. Проще говоря, <i>option</i> может находится в одном из двух конкретных состояний: <i>Some(x) </i>или <i>None</i></p><br/>
<pre><code class="scala">val anOptionInt: Option[Int] = Some(1)</code></pre><br/>
<p>или</p><br/>
<pre><code class="scala">val anOptionInt: Option[Int] = None</code></pre><br/>
<p>Можно оперировать с <i>option</i> при помощи функций isDefined и getOrElse (чтобы указать значение по умолчанию) но более часто используемая ситуация когда монады используется с операторами <i>map</i>, <i>foreach</i> или <i>fold</i>, для которых <i>option</i> представляет из себя коллекцию содержащую 0 или 1 элемент.</p><br/>
<p>Для примера можно подсчитать сумму двух опциональных переменных следующим образом:</p><br/>
<pre><code class="scala">val n1Option: Option[Int] = Some(1)
val n2Option: Option[Int] = Some(2)
val sum = for (n1 &lt;- n1Option; n2 &lt;- n2Option) yield {n1 + n2 }</code></pre><br/>
<p>В Переменной <i>sum</i> будет значение <i>Some(3)</i>. Наглядный пример того как <i>for</i> может быть использован как <i>foreach</i> или <i>flatMap</i> в зависимости от использования ключевого слова <i>yield</i>.</p><br/>
<p>Другой пример:</p><br/>
<pre><code class="scala">case class Person(name: String, age: Option[Int])
val person: Option[Person] = Some(Person("Jack", Some(1)))
for (p &lt;- person; age &lt;- p.age)  {
   println(s"The person is age $age")
}</code></pre><br/>
<p>Будет напечатана строчка "The person is age 1"</p><br/>
<h5><b>Подход в Kotlin</b></h5><br/>
<p>Kotlin заимствует синтаксис groovy, достаточно практичный в повседневном использовании. В Kotlin все типы <i>non-nullable</i> и должны быть в явном виде объявлены <i>nullable</i> с помощью "?" если они могут содержать <i>null</i>.</p><br/>
<p>Тот же пример может быть переписан следующим образом:</p><br/>
<pre><code class="scala">val n1: Int? = 1
val n2: Int? = 2
val sum = if (n1 != null &amp;&amp; n2 != null) n1 + n2 else null</code></pre><br/>
<p>Это намного ближе к Java синтаксису за исключением того, что Kotlin принудительно выполняет проверки во время компиляции, запрещая использовать <i>nullable</i> переменные без проверки на <i>null</i>, так что можно не бояться NullPointerException. Также нельзя присвоить <i>null</i> переменной объявленной как <i>non-nullable</i>. Помимо всего компилятор достаточно умный, чтобы избавить от повторной проверки переменной на <i>null</i>, что позволяет избежать многократной проверки переменных как в Java.</p><br/>
<p>Эквивалентный Kotlin код для второго примера будет выглядеть следующим образом:</p><br/>
<pre><code class="scala">data class Person(val name: String, val age: Int?)
val person: Person? = Person("Jack", 1)
if (person?.age != null) {
  printn("The person is age ${person?.age}")
}</code></pre><br/>
<p>Или альтернативный вариант с использованием "let", который заменает "if" блок на:</p><br/>
<pre><code class="scala">person?.age?.let {
  person("The person is age $it")
}</code></pre><br/>
<h4><b>Сравнение</b></h4><br/>
<p>Я предпочитаю подход в Kotlin. Он гораздо более читабельный и понятный, и проще разобраться что происходит в многократных вложенных уровнях. Подход Scala отталкивается от поведения монад, который конечно нравится некоторым людям, но по собственному опыту могу сказать, что код становится излишне перегруженным уже для небольших вложений. Существует огромное количество подводных камней у подобного усложнения в использовании <i>map</i> или <i>flatMap</i>, причем вы даже не получите предупреждение при компиляции, если вы делаете что-то не так в мешанине из монад или используя <i>pattern match</i> без поиска альтернативных вариантов, что в результате выливается в <i>runtime exception</i> которые не очевидны. </p><br/>
<p>Подход в Kotlin также уменьшает разрыв при интеграции с кодом Java благодаря тому что типы из нее по умолчанию nullable (<i>тут автор не совсем корректен. Типы из Java попадают в промежуточное состояние между nullable и not-nullable, которое в будущем можно уточнить</i>), тогда как Scala приходится поддерживать <i>null</i> как концепт без <i>null-safely</i> защиты. </p><br/>
<h4><b>Функциональные коллекции</b></h4><br/>
<p>Scala, конечно, поддерживает функциональный подход. Kotlin чуть в меньшей степени, но основные идеи поддерживаются.</p><br/>
<p>В примере ниже нет особых различий в работе <i>fold</i> и <i>map</i> функций:</p><br/>
<p><u>Scala</u></p><br/>
<pre><code class="scala">val numbers = 1 to 10
val doubles = numbers.map { _ * 2 }
val sumOfSquares = doubles.fold(0) { _ + _ }</code></pre><br/>
<p><u>Kotlin</u></p><br/>
<pre><code class="scala">val numbers = 1..10
val doubles = numbers.map { it * 2 }
val sumOfSquares = doubles.fold(0) {x,y -&gt; x+y }</code></pre><br/>
<p>Оба языка поддерживают концепт цепочки "ленивых" вычислений. Для примера вывод 10 четных чисел будет выглядеть следующим образом:</p><br/>
<p><u>Scala</u></p><br/>
<pre><code class="scala">val numbers = Stream.from(1)
val squares = numbers.map { x =&gt; x * x }
val evenSquares = squares.filter { _%2 == 0 }
println(evenSquares.take(10).toList)</code></pre><br/>
<p><u>Kotlin</u></p><br/>
<pre><code class="scala">val numbers = sequence(1) { it + 1 }
val squares = numbers.map { it * it }
val evenSquares = squares.filter { it%2 == 0 }
println(evenSquares.take(10).toList())</code></pre><br/>
<h4><b><i>Implicit</i> преобразования vs <i>extension</i> методы</b></h4><br/>
<p>Эта та область в которой Scala и Kotlin немного расходятся. </p><br/>
<h5><b>Подход в Scala</b></h5><br/>
<p>В Scala есть концепция <i>implicit</i> преобразований, которая позволяет добавлять расширенный функционал для класса благодаря автоматическому преобразованию к другому классу при необходимости. Пример объявления:</p><br/>
<pre><code class="scala">object Helpers {
  implicit class IntWithTimes(x: Int) {
    def times[A](f: =&gt; A): Unit = {
       for(i &lt;- 1 to x) {   
         f 
       }
    }
  }
}</code></pre><br/>
<p>Потом в коде можно будет использовать следующим образом :</p><br/>
<pre><code class="scala">import Helpers._
5.times(println("Hello"))</code></pre><br/>
<p>Это выведет "Hello" 5 раз. Работает это благодаря тому, что при вызове функции "times" (которая на самом деле не существует в Int) происходит автоматическая упаковка переменной в объект IntWithTimes, в котором и происходит вызов функции.</p><br/>
<h5><b>Подход в Kotlin</b></h5><br/>
<p>Kotlin использует для подобного функционала <i>extension</i> функции. В Kotlin для того, чтобы реализовать подобный функционал нужно объявить обычную функцию, только с префиксом в виде типа, для которая делается расширение.</p><br/>
<pre><code class="scala">fun Int.times(f: ()-&gt; Unit) {
  for (i in 1..this) {
    f()
  }
}</code></pre><br/>
<pre><code class="scala">5.times { println("Hello")}</code></pre><br/>
<h4><b>Сравнение</b></h4><br/>
<p>Подход Kotlin соответствует тому как я в основном использую данную возможность в Scala, с небольшим преимуществом в виде чуть более упрощенной и понятной записи.</p><br/>
<h4><b>Особенности Scala которых нет в Kotlin и по которым я не буду скучать</b></h4><br/>
<p>Одна из лучших особенностей Kotlin для меня даже не в том функционале что есть, а больше в том функционале которого нет в Kotlin из Scala.</p><br/>
<ul>
<li>Вызов по имени — Это разрушает читабельность. Если функция передается было бы гораздо легче увидеть что передается указатель на функции при простом просмотре кода. Я не вижу никаких преимуществ, которое это дает по сравнению с явной передачей лямбд.<br/>
</li>
<li>Implicit преобразования — Это то что я действительно ненавижу. Это приводит к ситуации, когда поведение кода значительно меняется в зависимости от того, что было импортировано. В результате действительно тяжело сказать какая переменная будет передана в функцию без хорошей поддержки IDE.<br/>
</li>
<li>Перегруженный <i>for</i> — Проблема с несколькими монадами, показанная выше. <br/>
</li>
<li>Беспорядок с опциональным синтаксисом <i>infix</i> и <i>postfix</i> операторов — Kotlin чуть более формализованный. В результате код в нем менее двухсмысленный,  его проще читать и не так легко простой опечатке стать неочевидной ошибкой. <br/>
</li>
<li>Переопределением операторов по максимуму — Kotlin разрешает переопределение только основных операторов (+, — и т.п.). Scala разрешает использовать любую последовательности символов. Действительно ли мне нужно знать разницу между "~%#&gt;" и "~+#&gt;"?<br/>
</li>
<li>Медленное время компиляции. <br/>
</li>
</ul><br/>
<p><i>Спасибо за внимание.<br/>
Оригинал <a href="https://agilewombat.com/2016/02/01/scala-vs-kotlin/">Scala vs Kotlin</a><br/>
P.S. В некоторые местах в переводе специально оставил слова без перевода (null, null safely, infix, postfix и т.п.).</i></p></div></div></div><!----><!----></div><!----><!----></div><!--]--><!----><div
