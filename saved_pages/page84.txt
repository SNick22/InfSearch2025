приложения (в особенности Spring Boot) на Kotlin. Основные ссылки на документацию: <a href="https://kotlinlang.ru/docs" rel="noopener noreferrer nofollow">Kotlin Docs</a> (на русском ссылки можно <a href="https://kotlinlang.ru/docs/" rel="noopener noreferrer nofollow">заменять на "ru"</a>,  у меня работает только чз VPN).</p><h2>Инициализация Gradle‑Kotlin проекта</h2><ul><li><p>Создаем новый Gradle‑Kotlin проект: <a href="https://kotlinlang.org/docs/jvm-create-project-with-spring-boot.html" rel="noopener noreferrer nofollow">Create a Spring Boot project with Kotlin</a>. <strong>Kts</strong> является расширением для <a href="https://sebhastian.com/kotlin-file-extension/" rel="noopener noreferrer nofollow">Kotlin скриптов</a>. Дополнительно: <a href="https://github.com/spring-guides/tut-spring-boot-kotlin#understanding-the-gradle-build" rel="noopener noreferrer nofollow">Understanding the Gradle Build</a>.</p></li><li><p>Меняем <a href="https://kotlinlang.org/docs/releases.html#release-details" rel="noopener noreferrer nofollow">версию Kotlin на последнюю</a> и перегружаем/обновляем проект</p></li><li><p>Копируем без изменения ресурсы <em>src/main/resources</em> и <em>src/test/resources</em> </p></li></ul><h3>Миграция базовых классов/интерфейсов</h3><blockquote><p>Преобразовывать классы Java в Kotlin в IDEA можно <a href="https://www.tutorialkart.com/kotlin/convert-java-file-to-kotlin-file" rel="noopener noreferrer nofollow">через конвертацию (Ctrl+Alt+Shift+K) или просто копируя Java код в Kotlin класс</a>. После этого обычно требуется ручные правки.</p></blockquote><ul><li><p>В Kotlin в одном файле можно создавать несколько публичных классов, поэтому, если классы небольшие, связанные между собой и их в проекте будет ограниченное количество, их помещяют в один общий с <a href="https://kotlinlang.org/docs/coding-conventions.html#source-file-names" rel="noopener noreferrer nofollow">обопщающим названием</a></p></li><li><p><a href="https://kotlinlang.org/docs/inheritance.html" rel="noopener noreferrer nofollow">Наследовать можно только открытые и абстрактные классы</a>, а по умолчанию Kotlin классы <code>final</code>. Все подклассы делаем <code>open</code></p></li><li><p>В Kotlin приходится заранее думать про <a href="https://kotlinlang.org/docs/null-safety.html" rel="noopener noreferrer nofollow">Nullable and non‑nullable types</a>. Чтобы не пропустить ошибку, после конвертации можно сначала убрать все <em>Nullable?</em> типы и затем добавлять <code>"?"</code> только там, где он действительно требуется. Аннотация <code>@NonNull</code> и <a href="https://kotlinlang.org/docs/null-safety.html#the-operator" rel="noopener noreferrer nofollow">!! operator</a> принудительного преобразования нулевого типа в ненулевой для non‑nullable типов не нужены.</p></li><li><p>Обычно, для улучшения читаемости и минимизации ошибок, если в конструкторае Kotlin несколько параметров, их принято распологать в отдельной строке: ставим курсор на любой из параметров и делаем <em>Alt+Enter→Put parameters on separate lines</em>. </p></li><li><p>Делаем везде, где возможно, <a href="https://kotlinlang.org/docs/functions.html#single-expression-functions" rel="noopener noreferrer nofollow">реализацию методов в одну строку</a>, тип возвращаемого значения, если он очевиден или неважен, опускаем и используем <a href="https://kotlinlang.org/docs/strings.html#string-templates" rel="noopener noreferrer nofollow">String templates</a> — очень удобную фичу вставки значений прямо в строку (особенно часто используется в <code>toString</code>)</p></li><li><p>Интерфейсы Kotlin <a href="https://kotlinlang.org/docs/interfaces.html#properties-in-interfaces" rel="noopener noreferrer nofollow">могут содержать свойства</a>. Мой базовый интерфейс <code>HasId</code> , от которого наследую все сущности JPA и все объекты DTO, выглядит так:</p></li></ul><pre><code class="kotlin">interface HasId {
    @get:Schema(accessMode = Schema.AccessMode.READ_ONLY)
    var id: Int?

    @JsonIgnore
    fun isNew() = id == null

    // doesn't work for hibernate lazy proxy
    fun id(): Int {
        Assert.notNull(id, "Entity must has id")
        return id!!
    }
}
</code></pre><ul><li><p>в <code>@Schema</code>для <em>springdoc</em> по умолчанию <a href="https://stackoverflow.com/questions/27900041/rest-put-ids-in-body-or-not/28025008#28025008" rel="noopener noreferrer nofollow">прячем id поля вводимых объектов</a> в Swagger. IDEA требует к ней <a href="https://kotlinlang.org/docs/annotations.html#annotation-use-site-targets" rel="noopener noreferrer nofollow">«get/set use‑site target»</a>.</p></li><li><p>В <code>id()</code> после проверки на null делается <a href="https://kotlinlang.org/docs/null-safety.html#the-operator" rel="noopener noreferrer nofollow">принудительное приведение к non‑nullable type</a>.</p></li></ul><h2>DTO</h2><ul><li><p>Классы TO будем делать <a href="https://kotlinlang.org/docs/data-classes.html" rel="noopener noreferrer nofollow">классами данных</a> (по сути это так и есть, кроме того удобно сразу иметь сгенерированные <code>equals()/hashCode()/copy()</code>). Для этого нам нужно объявлять все поля в конструкторе (синтаксис похоже на Java records, только поля будут изменяемые <code>var</code>). При этом базовые классы и их поля <a href="https://stackoverflow.com/a/55701197/548473" rel="noopener noreferrer nofollow">придется открывать </a><code>open</code>, а в наследуемых классах поля перекрывать <code>override</code>. Также, для <a href="https://kotlinlang.org/docs/classes.html#a2a48691" rel="noopener noreferrer nofollow">генерации конструктор без параметров</a>, задаем всем полям в конструкторах значения по умолчанию. Lombok в Kotlin нет, копируем Java код без его аннотаций и по <em>Alt+Enter</em> добавляем <code>import</code>.</p></li><li><p>Строковые <a href="https://kotlinlang.org/docs/null-safety.html" rel="noopener noreferrer nofollow">Non‑Null типы</a> инициализируем <code>=""</code>  </p></li><li><p>Чтобы не создавать лишних аннотации (только на поля, исключая геттеры и сеттеры) <a href="https://kotlinlang.org/docs/annotations.html#annotation-use-site-targets" rel="noopener noreferrer nofollow">делаем к ним use‑site targets указатели</a> <code>field:</code>. </p></li></ul><h3>Entities</h3><p>Entities классы <strong>не</strong> принято делать классами данных <a href="https://github.com/spring-guides/tut-spring-boot-kotlin#persistence-with-jpa" rel="noopener noreferrer nofollow">смотри Note</a> (напомню, что классы автоматически открываются у нас через <a href="https://github.com/spring-guides/tut-spring-boot-kotlin#plugins" rel="noopener noreferrer nofollow">plugin.jpa</a>). Кроме того в них, для правильной работы Hibernate, нельзя переопределять <code>equals()/hashcode()</code>, как это делают классы данных. К полям модели нужен доступ извне, делаем их <a href="https://stackoverflow.com/a/38328352/548473" rel="noopener noreferrer nofollow">public по умолчанию</a> (см.<a href="https://kotlinlang.org/docs/properties.html" rel="noopener noreferrer nofollow">свойства</a>). </p><ul><li><p>Переносим все поля в конструктор, добавляем к аннотациям <code>@field:</code>, для конструктора без параметров делаем инициализацию по умолчанию, код методов причесываем в <a href="https://kotlinlang.org/docs/coding-conventions.html#control-flow-statements" rel="noopener noreferrer nofollow">стиле Kotlin</a> (put short branches on the same line as the condition, without braces)</p></li><li><p>Мой базовый интерфейс <code>BaseEntity</code> выглядит так:</p></li></ul><pre><code class="kotlin">@MappedSuperclass
@Access(AccessType.FIELD)
abstract class BaseEntity(
    @field:Id
    @field:GeneratedValue(strategy = GenerationType.IDENTITY)
    override var id: Int? = null

) : HasId {
    //    https://stackoverflow.com/questions/1638723
    override fun equals(other: Any?): Boolean {
        if (this === other) return true
        if (other == null || javaClass != ProxyUtils.getUserClass(other)) return false
        val that = other as BaseEntity
        return id != null &amp;&amp; id == that.id
    }

    override fun hashCode() = id ?: 0
    override fun toString() = "${javaClass.simpleName}:$id"
}</code></pre><ul><li><p>Для <code>BaseEntity.hashCode</code> используем <a href="https://kotlinlang.org/docs/null-safety.html#elvis-operator" rel="noopener noreferrer nofollow">Elvis operator</a>.</p></li></ul><h2>Repositories</h2><ul><li><p>Дефолтные методы в <code>BaseRepository</code> Kotlin не видит, нужно <a href="https://stackoverflow.com/a/77298354/548473" rel="noopener noreferrer nofollow">добавить их поддержку</a>: опция <code>-Xjvm-default=all</code> и <code>@JvmDefaultWithCompatibility</code></p></li></ul><p>Мой базовый интерфейс, от которого <a href="https://stackoverflow.com/questions/42781264/multiple-base-repositories-in-spring-data-jpa" rel="noopener noreferrer nofollow">наследуются все репозитории</a>:</p><pre><code class="kotlin">@NoRepositoryBean
@JvmDefaultWithCompatibility
interface BaseRepository&lt;T&gt; : JpaRepository&lt;T, Int?&gt; {
    //    https://docs.spring.io/spring-data/jpa/docs/current/reference/html/#jpa.query.spel-expressions
    @Transactional
    @Modifying
    @Query("DELETE FROM #{#entityName} e WHERE e.id=:id")
    fun delete(id: Int): Int

    //  https://stackoverflow.com/a/60695301/548473 (existed delete code 204, not existed: 404)
    fun deleteExisted(id: Int) {
        if (delete(id) == 0) throw NotFoundException("Entity with id=$id not found")
    }

    fun getExisted(id: Int): T = findById(id).orElseThrow { NotFoundException("Entity with id=$id not found") }
}</code></pre><ul><li><p>Если мы не привязаны к <code>JpaRepository API</code> , заменяем Java Optional на Kotlin nullable, <a href="https://typealias.com/guides/java-optionals-and-kotlin-nulls/#orelsethrow" rel="noopener noreferrer nofollow">переучиваемся работать с nullable еще раз</a> :)</p></li></ul><h2>Залогиненный пользователь</h2><ul><li><p>Для получения авторизованного пользователя из любого места приложения вместо <a href="https://kotlinlang.org/docs/object-declarations.html#companion-objects" rel="noopener noreferrer nofollow">companion objects</a> сделал <a href="https://kotlinlang.org/docs/functions.html#function-scope" rel="noopener noreferrer nofollow">top-level functions</a>  </p></li><li><p>Для проверки в <code>authUser</code> используем <a href="https://www.baeldung.com/kotlin/preconditions" rel="noopener noreferrer nofollow">Preconditions</a></p></li><li><p>Коллизию имени с <em>org.springframework.security.core.userdetails.User</em>, разрешаем с помощью <a href="https://kotlinlang.org/docs/packages.html#imports" rel="noopener noreferrer nofollow">import as SecurityUser</a></p></li><li><p><code>AuthUser.user</code> при обновлении пользователя переприсваивается, делаем его <code>var</code>  </p></li></ul><pre><code class="kotlin">import org.springframework.security.core.userdetails.User as SecurityUser

fun safeAuthUser(): AuthUser? {
    val auth = SecurityContextHolder.getContext().authentication ?: return null
    val principal = auth.principal
    return if (principal is AuthUser) principal else null
}

fun authUser(): AuthUser = checkNotNull(safeAuthUser()) { "No authorized user found" }

class AuthUser(var user: User) : SecurityUser(user.email, user.password, user.roles) {
    fun id() = user.id()
    fun hasRole(role: Role) = user.hasRole(role)

    override fun toString() = "AuthUser:${user.id}[${user.email}]"
}</code></pre><h2>Логирование</h2><p>Наиболее красиво работать с логами через <a href="https://github.com/MicroUtils/kotlin-logging" rel="noopener noreferrer nofollow">kotlin‑logging</a>: подключаем <em>kotlin‑logging‑jvm</em></p><ul><li><p><a href="https://realjenius.com/2017/08/31/logging-in-kotlin/" rel="noopener noreferrer nofollow">Kotlin Logging Without the Fuss</a></p></li><li><p><a href="https://stackoverflow.com/questions/57728079/kotlin-best-way-to-log-using-slf4j" rel="noopener noreferrer nofollow">Kotlin best way to log</a></p></li></ul><h3>Бины Spring</h3><ul><li><p> Для <code>@Autowired</code> используем <a href="https://kotlinlang.org/docs/properties.html#late-initialized-properties-and-variables" rel="noopener noreferrer nofollow">отложенную инициализацию</a> (<code>lateinit var</code> )</p></li><li><p>Все методы, которые переопределяются и <strong>проксируется Spring</strong> должны быть <code>open</code> </p></li></ul><h3>Общие замечания</h3><ul><li><p>Используем, где возможно универсальное <a href="https://kotlinlang.org/docs/control-flow.html#when-expression" rel="noopener noreferrer nofollow">выражение when</a>, замену Java <code>Optional</code> на <a href="https://typealias.com/guides/java-optionals-and-kotlin-nulls/" rel="noopener noreferrer nofollow">Kotlin Nulls</a> и <a href="https://kotlinlang.org/docs/java-to-kotlin-interop.html#checked-exceptions" rel="noopener noreferrer nofollow">нет проверяемых исключений</a>, <code>@Throws</code> требуется только для вызова методов из Java </p></li><li><p>Kotlin позволяет <a href="https://kotlinlang.org/docs/extensions.html" rel="noopener noreferrer nofollow">расширять класс путём добавления нового функционала</a>, что позволяет делать код еще более объектно‑ориентированным (<a href="https://medium.com/@y9vad9/extension-oriented-design-6d9898c33e5f" rel="noopener noreferrer nofollow">Extension Oriented Design</a>). Старайтесь увидеть и применять этот дизайн везде, где к объекту добавляются новые свойства. Для всех расширений уровня проекта создал в корневом пакете <code>Extensions.kt</code>  </p></li><li><p>Kotlin по умолчанию генерирует getter/setter для всех полей класса. В случае его переопределения будет конфликт, решается через <a href="https://stackoverflow.com/a/62338482/548473" rel="noopener noreferrer nofollow">запрет на автогенерацию через аннотацию @JvmField</a>  </p></li><li><p>Заменяем Java классы <code>Class</code>  на Kotlin <a href="https://kotlinlang.org/api/latest/jvm/stdlib/kotlin.reflect/-k-class/" rel="noopener noreferrer nofollow">KClass</a> и Java коллекции на <a href="https://kotlinlang.org/docs/collections-overview.html" rel="noopener noreferrer nofollow">их Kotlin аналоги</a></p></li><li><p>Если в функции несколько аргументов, заданных по умолчанию, можно менять только определенные, используя <a href="https://kotlinlang.org/docs/functions.html#named-arguments" rel="noopener noreferrer nofollow">определенный именованный параметр</a> </p></li></ul><p>Напоследок еще раз — статья не предназначена для чтения. Это скорее набор практический правил из курса Spring Boot REST API приложение на Kotlin<strong> </strong>(в рамках наших курсов «Из Middle в Senior», см. предыдущий пост по курсу <strong>"</strong><a href="https://habr.com/ru/articles/765332/" rel="noopener noreferrer nofollow">Работа с документами в Java</a><strong>"</strong>)<strong> </strong>для миграции своего Java приложения (в особенности Spring Boot), поэтому обращайтесь к ней, когда решитесь на миграцию.</p><p>И да пребудет с вами сила:)!</p><p>PS: буду признателен за любые дополнения, замечания и комментарии к миграции кода на Kotlin.</p><p></p></div></div></div><!----><!----></div><!----><!----></div><!--]--><!----><div class="tm-article-presenter__meta" data-test-id="article-meta-links"><div class="tm-separated-list
