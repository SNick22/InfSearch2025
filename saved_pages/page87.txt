xmlns="http://www.w3.org/1999/xhtml"><a href="https://habrahabr.ru/company/redmadrobot/blog/301174/"><img src="https://habrastorage.org/r/w1560/files/311/e39/e6f/311e39e6f7c148118698f6c251141ec4.png" Эта статья про Generics в Kotlin — особенности их использования, сходства и различия с Generics в Java.<br/>
<br/>
<div class="spoiler"><b class="spoiler_title">В двух словах про Generics</b><div class="spoiler_text">Если коротко, то Generics — это способ сказать, что класс, интерфейс или метод будут работать не с каким-то конкретным типом, а просто с каким-то. С каким именно будет определено из контекста. Например:<br/>
<br/>
<pre><code class="java">public interface List&lt;E&gt; extends Collection&lt;E&gt; {
	//...
}</code></pre><br/>
Заранее неизвестно, объекты какого класса будут содержаться в списке, но это определится при его использовании:<br/>
<br/>
<pre><code class="java">List&lt;String&gt; list = new ArrayList&lt;&gt;();</code></pre><br/>
Теперь это не просто список, а список строк. Generics помогают обеспечить типобезопасность: в List можно попытаться положить любой объект, но в List&lt;String&gt; — только String или один из его потомков.</div></div><br/>
Я разделю рассказ про Generics на две части: собственно Generics и использование Wildcards. Пока речь не заходит про Wildcards, использование Generics в Kotlin мало чем отличается от Java.<br/>
<a name="habracut"></a><br/>
Те же generic-классы:<br/>
<br/>
<pre><code class="java">// Java
public class SomeGenericClass &lt;T&gt; {

	private T mSomeField;

    	public void setSomeField(T someData) {
        	mSomeField = someData;
    	}

    	public T getSomeField() {
        	return mSomeField;
    	}

}</code></pre><br/>
<pre><code>// Kotlin
class SomeGenericClass &lt;T&gt; {

    	private var mSomeField: T? = null

    	fun setSomeField(someData: T?) {
        	mSomeField = someData
    	}

    	fun getSomeField(): T? {
        	return mSomeField
    	}
    
}</code></pre><br/>
Те же generic-методы:<br/>
<br/>
<pre><code class="java">// Java
public &lt;K&gt; K makeSomething(K someData) {
    	K localData = someData;
    	//...
    	return localData;
}</code></pre><br/>
<pre><code>// Kotlin
fun &lt;K&gt; makeSomething(someData : K) : K {
    	var localData = someData
    	//...
    	return localData
}</code></pre><br/>
Generics могут быть дополнительно ограничены и в Java:<br/>
<br/>
<pre><code class="java">// Java
public &lt;K extends Number&gt; K makeSomething(K someData) {
    	K localData = someData;
    	//...
    	return localData;
}</code></pre><br/>
И в Kotlin:<br/>
<br/>
<pre><code>// Kotlin
fun &lt;K : Number&gt; makeSomething(someData : K) : K {
    	var localData = someData
    	//...
    	return localData
}</code></pre><br/>
Такие ограничения обозначают, что вместо K может быть использован не любой класс, а только удовлетворяющий условию (в данном случае — Number или класс, его наследующий).<br/>
<br/>
<pre><code class="java">//корректно
makeSomething(1)

//некорректно
makeSomething(“string”)</code></pre><br/>
Ограничения могут быть и комплексными, например, показывающими, что передаваемый в метод объект должен наследовать какой-то класс и реализовывать какой-то интерфейс, например:<br/>
<br/>
<pre><code class="java">//Java
public static &lt;T extends Interaction &amp; Fragment&gt; SomeFragment newInstance(T interactor) {
    	SomeFragment fragment = new SomeFragment();
    	fragment.setTargetFragment(interactor, 0);
    	return fragment;
}</code></pre><br/>
<pre><code>//Kotlin
fun &lt;T&gt; newInstance(interactor : T) : SomeFragment where T : Interaction, T : Fragment {
    	val fragment = SomeFragment()
    	fragment.setTargetFragment(interactor, 0)
    	return fragment
}</code></pre><br/>
Обратите внимание, что в Kotlin для комплексных ограничений используется другой синтаксис: добавилось немного синтаксического сахара. Можно не указывать параметр типа, если он может быть определен по контексту:<br/>
<br/>
<pre><code>// Kotlin
val someGenericClassInstance = SomeGenericClass("This is String")</code></pre><br/>
А в Java придется:<br/>
<br/>
<pre><code class="java">// Java
SomeGenericClass&lt;String&gt; someGenericClassInstance = new SomeGenericClass&lt;&gt;("This is String");</code></pre><br/>
Таким образом, главное, что нужно знать про Generics при переходе с Java на Kotlin — делайте все так же, как делали в Java. Попытки сделать что-то по-новому, “по-котлиновски”, скорее всего приведут только к новым сложностям.<br/>
<br/>
<h2>Wildcards</h2><br/>
Перейдем ко второй части. Wildcards — особый случай, вызывающий больше всего сложностей и в Kotlin, и в Java. Основная проблема Generics — их инвариантность: List&lt;String&gt; не является потомком List&lt;Object&gt;. В противном случае могли бы происходить ошибки вида:<br/>
<br/>
<pre><code class="java">//Java
List&lt;String&gt; strs = new ArrayList&lt;String&gt;();
List&lt;Object&gt; objs = strs;
//objs - List&lt;Object&gt;, так что туда можно положить Integer
objs.add(1);
//но strs - List&lt;String&gt;, так что get() должен возвращать String
String s = strs.get(0);</code></pre><br/>
Инвариантность Generics позволяет не допустить этого, но, с другой стороны, вносит дополнительные ограничения. Так, при использовании обычных Generics невозможно List&lt;String&gt; передать в метод, ожидающий в качестве параметра List&lt;Object&gt;. Во многих случаях удобно иметь такую возможность.<br/>
<br/>
Wildcards позволяют разрешить такое поведение, обозначив, что в данном месте ожидается какой-то параметр типа, но не какой-то конкретный. При этом Wildcards тоже могут быть особым образом ограничены, что разбивает вопрос на 3 части: <br/>
<ul>
<li>неограниченные Wildcards</li>
<li>ковариантные Wildcards</li>
<li>контравариантные Wildcards</li>
</ul><br/>
<br/>
<i>Ограниченные “снизу” ковариантные Wildcards</i> используются в случаях, когда ожидается generic-класс от какого-то класса или его потомков. Например:<br/>
<br/>
<pre><code class="java">// Java
public interface Container&lt;T&gt; {
    	T getData();
    	void putData(T data);
}

static void hideView(Container&lt;? extends View&gt; viewContainer) {
    	viewContainer.getData().setVisibility(View.GONE);
}</code></pre><br/>
Здесь метод hideView ожидает объект, реализующий интерфейс Container, но не любой и не только содержащий View, а содержащий View или какой-либо другой класс, наследующий View. Это и называется ковариантностью.<br/>
<br/>
В Kotlin такое поведение может быть реализовано похожим образом:<br/>
<br/>
<pre><code>// Kotlin
interface Container&lt;T&gt; {
   	fun getData() : T
   	fun putData(data : T)
}

fun hideView (viewContainer : Container&lt;out View&gt;) {
   	viewContainer.getData().visibility = View.GONE;
}</code></pre><br/>
При этом на использование параметра, объявленного как Wildcards, накладываются дополнительные ограничения.<br/>
<br/>
В Java ковариантные Wildcards можно использовать для получения данных без ограничений, при этом возвращаться данные будут в соответствии с обозначенной границей (в примере выше getData() вернет View, даже если на самом деле контейнер содержал TextView). Но положить в него нельзя ничего, кроме null, иначе бы это вызвало те же проблемы, что возникли бы у Generics, не будь они инвариантными.<br/>
<br/>
<pre><code class="java">//Java
static void hideView(Container&lt;? extends View&gt; viewContainer) {
    	//getData() вернет View даже если на самом деле в контейнере содержится, например, TextView
    	viewContainer.getData().setVisibility(View.GONE);
    	//положить внутрь можно только null, так как неизвестно, какой именно класс должен содержаться в этом контейнере
    	//такой вызов корректен
    	viewContainer.putData(null);
    	//а такой - некорректен, потому что на самом деле это может быть, например, Container&lt;ImageView&gt;
    	viewContainer.putData(new View(App.getContext()));
}</code></pre><br/>
В Kotlin ограничения почти такие же. Из-за особенностей типов в Kotlin положить внутрь такого параметра нельзя даже null. Ключевое слово out отлично описывает происходящее.<br/>
<br/>
<pre><code>//Kotlin
fun hideView (viewContainer : Container&lt;out View&gt;) {
   	viewContainer.getData().visibility = View.GONE;
	//некорректен даже такой вызов, потому что неизвестно, содержит контейнер View (или какого-то из его потомков) или View?
	viewContainer.putData(null)
}</code></pre><br/>
<i>Ограниченные “сверху” контравариантные Wildcards</i> используются для обозначения мест, где ожидается generic-класс от какого-то класса или его предков. Традиционный пример контравариантных Wildcards — компараторы:<br/>
<br/>
<pre><code class="java">// Java
public static &lt;T&gt; void sort(List&lt;T&gt; list, Comparator&lt;? super T&gt; comparator) {
	//...
}</code></pre> <br/>
Допустим, в метод в качестве первого параметра передан List&lt;String&gt;, а в качестве второго — компаратор от любого из предков String, например, CharSequence: Comparator&lt;CharSequence&gt;. Так как String является потомком CharSequence, любые поля и методы, необходимые компаратору, будут и у объектов класса String:<br/>
<br/>
<pre><code class="java">//Java
class LengthComparator implements Comparator&lt;CharSequence&gt; {

	@Override
	public int compare(CharSequence obj1, CharSequence obj2) {
		//с объектами класса String в качестве параметров тоже будет работать
		if (obj1.length() == obj2.length()) return 0;
		if (obj1.length() &lt; obj2.length()) return -1;
		return 1;
   	}

}</code></pre><br/>
В Kotlin реализация аналогична:<br/>
 <br/>
<pre><code>// Kotlin
fun &lt;T&gt; sort(list : List&lt;T&gt;, comparator: Comparator&lt;in T&gt;) {
   	//…
}</code></pre><br/>
Есть у контравариантных Wildcards и вполне ожидаемые ограничения: считать значение из таких Wildcards можно, но возвращаться будет Object в Java и Any? в Kotlin.<br/>
<br/>
На этом этапе повторюсь: переходя с Java на Kotlin, следует делать все так же, как и делали. Хоть в официальной документации про Wildcards и написано “Kotlin doesn’t have any” (“В Kotlin их нет”), предлагаемый взамен механизм type projections (рассмотренный выше) во всех привычных случаях работает аналогично, никакие новые подходы не требуются.<br/>
<br/>
Но не обошлось и без новшеств. Кроме type projections, полностью аналогичного привычной модели Wildcards в Java, Kotlin предлагает еще один механизм — declaration-side variance. <br/>
<br/>
В случае если заранее известно, что generic-класс будет использоваться только как ковариантный (или только как контравариантный), указать это можно во время написания generic-класса, а не в момент его использования. В качестве примера опять же подойдут компараторы. Переписанный на Kotlin, java.util.Comparator мог бы выглядеть так:<br/>
<br/>
<pre><code>// Kotlin
interface Comparator&lt;in T&gt; {
   	fun compare(lhs: T, rhs: T): Int
   	override fun equals(other : Any?): Boolean
}</code></pre><br/>
И тогда его использование будет выглядеть следующим образом:<br/>
<br/>
<pre><code>// Kotlin
fun &lt;T&gt; sort(list : List&lt;T&gt;, comparator: Comparator&lt;T&gt;) {
   	//…
}</code></pre><br/>
При этом ограничения на использование параметра comparator будут такие же, как если бы &lt;in T&gt; было указано не на стороне декларации интерфейса, а на стороне его использования.<br/>
<br/>
Аналогичным образом при декларации класса может быть определено и ковариантное поведение.<br/>
<br/>
Последний не разобранный случай — <i>Wildcards без ограничений</i>. Такие, очевидно, используются в случаях, когда подходит generic от любого класса:<br/>
<br/>
<pre><code class="java">// Java
public interface Container&lt;T&gt; {
   	T getData();
   	void putData(T data);
}

static boolean isNull(Container&lt;?&gt; container) {
 	return container.getData() != null;
}</code></pre><br/>
В Kotlin аналогичный механизм называется star-projection. Во всех тривиальных случаях единственное его отличие от неограниченных Wildcards в Java — использование символа “*” вместо “?”:<br/>
<br/>
<pre><code>// Kotlin
interface Container&lt;T&gt; {
   	fun getData() : T
   	fun putData(data : T);
}

fun isNull(container : Container&lt;*&gt;) : Boolean {
   	return container.getData() != null;
}</code></pre><br/>
В Java неограниченные Wildcards используются по следующим правилам: положить в них можно только null, считывается всегда Object. В Kotlin положить внутрь нельзя ничего, а считывается объект класса Any?..<br/>
<br/>
При совместном использовании declaration-side variance и star-projection нужно учитывать, что ограничения суммируются. Так, при использовании контравариантного declaration-side variance (позволяющего положить внутрь что угодно, но считать только Any?) вместе со star-projection положить внутрь нельзя будет ничего (ограничение star-projection), а возвращаться будет все тот же Any? (в этом их ограничения совпадают).<br/>

Это и называется ковариантностью.\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\nВ Kotlin такое поведение может быть реализовано похожим образом:\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Cpre\u003E\u003Ccode\u003E\u002F\u002F Kotlin\ninterface Container&lt;T&gt; {\n   \tfun getData() : T\n   \tfun putData(data : T)\n}\n\nfun hideView (viewContainer : Container&lt;out View&gt;) {\n   \tviewContainer.getData().visibility = View.GONE;\n}\u003C\u002Fcode\u003E\u003C\u002Fpre\u003E\u003Cbr\u002F\u003E\nПри этом на использование параметра, объявленного как Wildcards, накладываются дополнительные ограничения.\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\nВ Java ковариантные Wildcards можно использовать для получения данных без ограничений, при этом возвращаться данные будут в соответствии с обозначенной границей (в примере выше getData() вернет View, даже если на самом деле контейнер содержал TextView). Но положить в него нельзя ничего, кроме null, иначе бы это вызвало те же проблемы, что возникли бы у Generics, не будь они инвариантными.\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Cpre\u003E\u003Ccode class=\"java\"\u003E\u002F\u002FJava\nstatic void hideView(Container&lt;? extends View&gt; viewContainer) {\n    \t\u002F\u002FgetData() вернет View даже если на самом деле в контейнере содержится, например, TextView\n    \tviewContainer.getData().setVisibility(View.GONE);\n    \t\u002F\u002Fположить внутрь можно только null, так как неизвестно, какой именно класс должен содержаться в этом контейнере\n    \t\u002F\u002Fтакой вызов корректен\n    \tviewContainer.putData(null);\n    \t\u002F\u002Fа такой - некорректен, потому что на самом деле это может быть, например, Container&lt;ImageView&gt;\n    \tviewContainer.putData(new View(App.getContext()));\n}\u003C\u002Fcode\u003E\u003C\u002Fpre\u003E\u003Cbr\u002F\u003E\nВ Kotlin ограничения почти такие же. Из-за особенностей типов в Kotlin положить внутрь такого параметра нельзя даже null. Ключевое слово out отлично описывает происходящее.\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Cpre\u003E\u003Ccode\u003E\u002F\u002FKotlin\nfun hideView (viewContainer : Container&lt;out View&gt;) {\n   \tviewContainer.getData().visibility = View.GONE;\n\t\u002F\u002Fнекорректен даже такой вызов, потому что неизвестно, содержит контейнер View (или какого-то из его потомков) или View?\n\tviewContainer.putData(null)\n}\u003C\u002Fcode\u003E\u003C\u002Fpre\u003E\u003Cbr\u002F\u003E\n\u003Ci\u003EОграниченные “сверху” контравариантные Wildcards\u003C\u002Fi\u003E используются для обозначения мест, где ожидается generic-класс от какого-то класса или его предков. Традиционный пример контравариантных Wildcards — компараторы:\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Cpre\u003E\u003Ccode class=\"java\"\u003E\u002F\u002F Java\npublic static &lt;T&gt; void sort(List&lt;T&gt; list, Comparator&lt;? super T&gt; comparator) {\n\t\u002F\u002F...\n}\u003C\u002Fcode\u003E\u003C\u002Fpre\u003E \u003Cbr\u002F\u003E\nДопустим, в метод в качестве первого параметра передан List&lt;String&gt;, а в качестве второго — компаратор от любого из предков String, например, CharSequence: Comparator&lt;CharSequence&gt;. Так как String является потомком CharSequence, любые поля и методы, необходимые компаратору, будут и у объектов класса String:\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Cpre\u003E\u003Ccode class=\"java\"\u003E\u002F\u002FJava\nclass LengthComparator implements Comparator&lt;CharSequence&gt; {\n\n\t@Override\n\tpublic int compare(CharSequence obj1, CharSequence obj2) {\n\t\t\u002F\u002Fс объектами класса String в качестве параметров тоже будет работать\n\t\tif (obj1.length() == obj2.length()) return 0;\n\t\tif (obj1.length() &lt; obj2.length()) return -1;\n\t\treturn 1;\n   \t}\n\n}\u003C\u002Fcode\u003E\u003C\u002Fpre\u003E\u003Cbr\u002F\u003E\nВ Kotlin реализация аналогична:\u003Cbr\u002F\u003E\n \u003Cbr\u002F\u003E\n\u003Cpre\u003E\u003Ccode\u003E\u002F\u002F Kotlin\nfun &lt;T&gt; sort(list : List&lt;T&gt;, comparator: Comparator&lt;in T&gt;) {\n   \t\u002F\u002F…\n}\u003C\u002Fcode\u003E\u003C\u002Fpre\u003E\u003Cbr\u002F\u003E\nЕсть у контравариантных Wildcards и вполне ожидаемые ограничения: считать значение из таких Wildcards можно, но возвращаться будет Object в Java и Any? в Kotlin.\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\nНа этом этапе повторюсь: переходя с Java на Kotlin, следует делать все так же, как и делали. Хоть в официальной документации про Wildcards и написано “Kotlin doesn’t have any” (“В Kotlin их нет”), предлагаемый взамен механизм type projections (рассмотренный выше) во всех привычных случаях работает аналогично, никакие новые подходы не требуются.\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\nНо не обошлось и без новшеств. Кроме type projections, полностью аналогичного привычной модели Wildcards в Java, Kotlin предлагает еще один механизм — declaration-side variance. \u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\nВ случае если заранее известно, что generic-класс будет использоваться только как ковариантный (или только как контравариантный), указать это можно во время написания generic-класса, а не в момент его использования. В качестве примера опять же подойдут компараторы. Переписанный на Kotlin, java.util.Comparator мог бы выглядеть так:\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Cpre\u003E\u003Ccode\u003E\u002F\u002F Kotlin\ninterface Comparator&lt;in T&gt; {\n   \tfun compare(lhs: T, rhs: T): Int\n   \toverride fun equals(other : Any?): Boolean\n}\u003C\u002Fcode\u003E\u003C\u002Fpre\u003E\u003Cbr\u002F\u003E\nИ тогда его использование будет выглядеть следующим образом:\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Cpre\u003E\u003Ccode\u003E\u002F\u002F Kotlin\nfun &lt;T&gt; sort(list : List&lt;T&gt;, comparator: Comparator&lt;T&gt;) {\n   \t\u002F\u002F…\n}\u003C\u002Fcode\u003E\u003C\u002Fpre\u003E\u003Cbr\u002F\u003E\nПри этом ограничения на использование параметра comparator будут такие же, как если бы &lt;in T&gt; было указано не на стороне декларации интерфейса, а на стороне его использования.\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\nАналогичным образом при декларации класса может быть определено и ковариантное поведение.\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\nПоследний не разобранный случай — \u003Ci\u003EWildcards без ограничений\u003C\u002Fi\u003E. Такие, очевидно, используются в случаях, когда подходит generic от любого класса:\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Cpre\u003E\u003Ccode class=\"java\"\u003E\u002F\u002F Java\npublic interface Container&lt;T&gt; {\n   \tT getData();\n   \tvoid putData(T data);\n}\n\nstatic boolean isNull(Container&lt;?&gt; container) {\n \treturn container.getData() != null;\n}\u003C\u002Fcode\u003E\u003C\u002Fpre\u003E\u003Cbr\u002F\u003E\nВ Kotlin аналогичный механизм называется star-projection. Во всех тривиальных случаях единственное его отличие от неограниченных Wildcards в Java — использование символа “*” вместо “?”:\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Cpre\u003E\u003Ccode\u003E\u002F\u002F Kotlin\ninterface Container&lt;T&gt; {\n   \tfun getData() : T\n   \tfun putData(data : T);\n}\n\nfun isNull(container : Container&lt;*&gt;) : Boolean {\n   \treturn container.getData() != null;\n}\u003C\u002Fcode\u003E\u003C\u002Fpre\u003E\u003Cbr\u002F\u003E\nВ Java неограниченные Wildcards используются по следующим правилам: положить в них можно только null, считывается всегда Object. В Kotlin положить внутрь нельзя ничего, а считывается объект класса Any?..\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\nПри совместном использовании declaration-side variance и star-projection нужно учитывать, что ограничения суммируются.
