<a name="habracut"></a><br/>
<br/>
<h2>1. Переопределение Java-методов в Kotlin-коде</h2><br/>
На тему взаимодействия Kotlin с Java-кодом сломано немало копий, выскажусь и я.<br/>
Одной из главных фич языка является то, что nullability объектов внутри Kotlin-кода проверяется уже на этапе компиляции. То есть, если попробовать обратиться к полю/методу объекта (через оператор <code>.</code>), объявленного как nullable (или передать его как аргумент в функцию, принимающую на вход notnull-переменную) — то такой код даже не скомпилируется. Чтобы компиляция прошла, Вам придётся воспользоваться оператором безопасного вызова <code>?.</code> — или же добавить для такого объекта явную проверку на null.<br/>
<br/>
Такой подход действительно хорошо защищает нас от <abbr title="Null Pointer Exception">NPE</abbr>. Однако, если вы обращаетесь к объекту, «пришедшему» из Java-кода — то этот подход не применяется:<br/>
<blockquote><abbr title="Любая ссылка в Java может принимать значение null, что делает требования Kotlin по null-safety непрактичными для объектов приходящих из Java. Типы, декларируемые в Java обрабатываются по-особому в Kotlin и называются платформенными типами. Null-проверки для таких типов являются менее строгими, таким образом безопасность для них гарантирована таким же образом, как и в Java.">Any reference in Java may be null, which makes Kotlin's requirements of strict null-safety impractical for objects coming from Java. Types of Java declarations are treated in Kotlin in a specific manner and called platform types. Null-checks are relaxed for such types, so that safety guarantees for them are the same as in Java.</abbr></blockquote> Таким образом, в Kotlin-коде необходимо обрабатывать все объекты из Java-кода как nullable. Всегда и везде, иначе есть риск получить NPE. Даже если от этого код будет выглядеть менее красивым, зато он будет более надёжным.<br/>
<br/>
Наверное, об этом знают уже все, кто пишет на Kotlin. Но наиболее ярко проблема проявляется при переопределении методов Java-класса в Kotlin-классе. Дело в том, что <abbr title="в данном случае Intellij IDEA или Android Studio">IDE</abbr> при вводе слова <code>override</code> заботливо предлагает авто-дополнение со списком доступных методов родительского класса. И nullability аргументов каждого метода (равно как и его возвращаемого значения) IDE проставляет, исходя из наличия аннотации <code><abbr title="здесь подразумеваются org.jetbrains.annotations.Nullable, javax.annotation.Nullable и т.п.">@Nullable</abbr></code>: если она указана для аргумента, то используется nullable-тип (т.е. с «вопросительным знаком») — иначе, используется «обычный» тип.<br/>
<br/>
То есть, если для объекта в Java-коде не проставлена ни одна из аннотаций <code>@Nullable/@NotNull</code> — то в Kotlin-коде по умолчанию для этого объекта будет использован тип без «вопросительного знака», и при обращении к его полям/методам мы сможем получить NPE. Но на самом деле, даже если в Java-коде использовалось <code>@NotNull</code> — то всё равно лучше не полагаться на IDE и самим добавить «вопросительный знак» к Kotlin-типу, и обрабатывать его как nullable. Почему так? Потому что, если в дальнейшем кто-то захочет расширить функционал Java-метода, добавив обработку <i>null</i>-значения, и уберёт аннотацию <code>@NotNull</code> (не проставив <code>@Nullable</code>) — то это никак не отразится на компиляции Kotlin-кода, и в нём опять возникнет риск получить NPE в рантайме…<br/>
<br/>
Данная проблема может возникнуть, к примеру, при обновлении какого-то утилитарного Java-пакета — и её сложно заметить. Поэтому лучше не доверять IDE в вопросе nullability типов, и явно обрабатывать в Kotlin-коде типы <b>всех</b> объектов, пришедших из Java, как nullable.<br/>
<br/>
<h2>2. Использование <i>delay()</i> внутри <i>synchronized</i>-методов</h2><br/>
В старой доброй Java почти не было разницы между подходами, когда:<br/>
<br/>
<ul>
<li>всё тело метода обёрнуто в <code>synchronized</code>-блок</li>
<li>сам метод объявлен с ключевым словом <code>synchronized</code></li>
</ul><br/>
То есть, разница была, но она касалась лишь производительности, а также использования заблокированного объекта другими потоками. Но результат исполнения для обоих подходов был одинаков.<br/>
<br/>
В Kotlin по-прежнему можно использовать <code>synchronized</code>-блоки из Java («первый» подход), а вот для «второго» подхода вместо ключевого слова <code>synchronized</code> нужно использовать одноимённую аннотацию <code>@Synchronized</code>. Однако при использовании Kotlin-корутин, вызывающих <code>delay()</code>, можно столкнуться с неожиданной разницей для вышеуказанных подходов. Это иллюстрирует следующий код:<br/>
<br/>
<pre><code class="kotlin">val lockObject = Object()

suspend fun methodUsingSynchronizedBlock()
{
    synchronized(lockObject)
    {
        println("before methodUsingSynchronizedBlock")
        Thread.sleep(5)
        println("after methodUsingSynchronizedBlock")
    }
}

@Synchronized
suspend fun fullySynchronizedMethod()
{
    println("before fullySynchronizedMethod")
    Thread.sleep(5)
    println("after fullySynchronizedMethod")
}

fun main()
{    
    repeat(2) {
        GlobalScope.launch { methodUsingSynchronizedBlock() }
    }
}</code></pre><br/>
После запуска данного кода, в консоль выведется:<br/>
<br/>

 after methodUsingSynchronizedBlock<br/>
 before methodUsingSynchronizedBlock<br/>
 after methodUsingSynchronizedBlock<br/>
</div>
                    </div><br/>
Что вполне соответствует ожидаемому.<br/>
<br/>
Что же будет, если внутри <code>main()</code> заменить <code>methodUsingSynchronizedBlock()</code> на <code>fullySynchronizedMethod()</code>?<br/>
<br/>
<div class="spoiler" role="button" tabindex="0">
                        <b class="spoiler_title">Результат</b>

Это тоже соответствует ожидаемому и вполне логично. Потоки работают по очереди, как мы и хотели, т.е. подходы ведут себя одинаково.<br/>
<br/>
Пока всё хорошо. Но что будет, если мы решим воспользоваться всей мощью Kotlin и <i>приостановить</i> каждую корутину, не блокируя весь поток? Т.е. заменим в обоих методах вызовы <code>Thread.sleep(5)</code> на <code>delay(5)</code> и посмотрим, что будет:<br/>
<br/>
<div class="spoiler" role="button" tabindex="0">
                        <b class="spoiler_title">Результат</b>

                    </div><br/>
То есть, мы имеем ошибку компиляции, которая сама по себе вполне логична — ведь критическая секция должна блокировать поток, а функция <code>delay()</code> этого не обеспечивает, поэтому компилятор запрещает нам вызывать её внутри критической секции.<br/>
<br/>
Но ошибка, как говорит нам компилятор, касается лишь метода <code>methodUsingSynchronizedBlock</code>! А что будет, если мы удалим этот метод (и оставим внутри <code>main()</code> вызов <code>fullySynchronizedMethod</code> вместо <code>methodUsingSynchronizedBlock</code>)?<br/>
<br/>
<div class="spoiler" role="button" tabindex="0">
                        <b class="spoiler_title">Исходный код</b>
                        <div class="spoiler_text"><pre><code class="kotlin">@Synchronized
suspend fun fullySynchronizedMethod()
{
    println("before fullySynchronizedMethod")
    delay(5)
    println("after fullySynchronizedMethod")
}

fun main()
{    
    repeat(2) {
        GlobalScope.launch { fullySynchronizedMethod() }
    }

</div>
                    </div><br/>
<div class="spoiler" role="button" tabindex="0">

 after fullySynchronizedMethod<br/>

Получается, что если весь метод объявлен с аннотацией <code>@Synchronized</code> — то использование <code>delay()</code> для него разрешается! И оно приводит к тому, что оба потока заходят внутрь критической секции — т.е. второй заходит в неё тогда, когда первый выполнил <code>delay()</code>, но ещё не успел выполнить завершающий <code>println()</code>. И получается весьма неожиданный результат…<br/>
<br/>
Способов избежать данной проблемы много. Можно явно блокировать поток через <code>Thread.sleep()</code>, можно оборачивать в <code>synchronized</code> лишь блок кода, а можно <a href="https://kotlinlang.org/docs/shared-mutable-state-and-concurrency.html#mutual-exclusion" rel="nofollow noopener noreferrer"><u>воспользоваться мьютексами</u></a> — в отличие от <code>synchronized</code>, они корректно работают с <code>delay()</code> в корутинах (и результат будет правильный, т.е. как в случае с <code>Thread.sleep()</code>).<br/>
<br/>
<h2>3. Вызовы <i>getter</i>'ов из Java-кода, выглядящих как поля класса</h2><br/>
Одна из фич Kotlin (являющаяся, как и большинство других, синтаксическим сахаром) — это <i>"<abbr title="по крайней мере, так их зовёт Android Studio">synthetic properties</abbr>"</i>, то есть возможность обращения к уже существующим геттерам/сеттерам Java-класса так, как будто вы обращаетесь к полям этого класса. Иначе говоря, Kotlin (неявно) автоматически генерирует эти properties для всех подходящих Java-классов.<br/>
<br/>
<div class="spoiler" role="button" tabindex="0">
                        <b class="spoiler_title">Простой пример использования synthetic properties</b>
                        <div class="spoiler_text">Возьмём стандартный Android-класс <i>android.view.View</i>, в котором уже имеется поле <i>mTag</i> и соответствующие ему геттер/сеттер:<br/>
<pre><code class="java">public class View implements Drawable.Callback, KeyEvent.Callback
{
    protected Object mTag = null;
    ...
    public Object getTag()  { return mTag; }
    public void setTag(final Object tag)  { mTag = tag; }
}</code></pre><br/>
Теперь, если при разработке на Kotlin мы захотим узнать значение <i>mTag</i> для какого-то объекта этого класса, то это можно будет сделать двумя способами:<br/>
<pre><code class="kotlin">val myView: android.view.View = /* какой-то инициализатор */
...
val myTag1 = myView.getTag()  // старый способ с использованием геттера
val myTag2 = myView.tag  // новый способ с использованием synthetic properties </code></pre><br/>
Таким образом, Kotlin даёт нам удобную возможность обращаться к <i>mTag</i> так, как будто мы обращаемся к полю с именем <i>tag</i> (это имя берётся из имён геттера/сеттера). Оба этих способа совершенно эквивалентны — но «старый» способ не нравится Android Studio, и она выдаст предупреждение при его использовании.<br/>
</div>
                    </div><br/>
Казалось бы, простая и удобная фича — что же в ней может быть опасного? А то, что геттер может быть не тривиальный (т.е. не вида <i>«return value»</i>), а довольно <b>сложный</b> — то есть, выполняться продолжительное время. Что может привести к проблемам при использовании подобного кода (пример для класса <i>FirebaseDatabase</i>, входящего состав Firebase):<br/>
<br/>
<pre><code class="kotlin">val dbTableNames = arrayOf("first", "second", "third", "fourth")
...

fun clearAllDbTables(db: FirebaseDatabase)
{
    for (currentName in dbTableNames)
        db.reference.child(currentName).setValue(null)
}</code></pre><br/>
С первого взгляда, в этом коде не видно каких-либо проблем. Но на самом деле, поле <i>reference</i> объекта <i>db</i> — это не поле, а вызов геттера <i>getReference()</i> класса <i>FirebaseDatabase</i>, который не просто возвращает нам уже существующий объект класса <i>DatabaseReference</i>, а каждый раз создаёт его заново.<br/>
<br/>
То есть, в цикле <i>for</i> поочерёдно будут созданы 4 одинаковых объекта, что плохо — хотя мы могли бы избежать этого, если бы просто вынесли получение <i>reference</i> за пределы цикла:<br/>
<br/>
<pre><code class="kotlin">fun clearAllDbTables(db: FirebaseDatabase)
{
    val rootReference = db.reference
    
    for (currentName in dbTableNames)
        rootReference.child(currentName).setValue(null)
}</code></pre><br/>
В общем, в первом варианте кода налицо проблема с производительностью из-за того, что геттер <i>getReference()</i> не тривиальный. Способ избежать проблемы очевиден — нужно проверять, является ли геттер тривиальным, и избегать его множественного вызова в противном случае.<br/>
<br/>
Наверняка кто-то возразит — «постойте, ведь в других языках программирования properties тоже есть, и никто не жалуется!».<br/>
<br/>
<div class="spoiler" role="button" tabindex="0">
                        <b class="spoiler_title">Да, это так, но в случае с Java и Kotlin есть пара нюансов</b>
                        <div class="spoiler_text"><ul>
<li>во-первых, в Java нет properties, поэтому при кодинге на Kotlin привыкший к Java программист может просто не знать о том, что при этом производится вызов геттера;</li>
<li>во-вторых, Kotlin всячески подталкивает программиста к использованию полей вместо методов при разработке собственного класса. Например, если в класс нужно добавить поле, доступное сторонним классам только на чтение, то рекомендуется делать это как:<br/>
<br/>
<pre><code class="kotlin">
var fieldName: FieldType
    private set</code></pre><br/>
вместо объявления private-поля <i>«fieldName»</i> с публичным тривиальным геттером. Таким образом, в Kotlin мы намного чаще обращаемся к полям класса напрямую (т.е. не через геттеры), чем в Java. И в итоге программист <b>привыкает</b> к этому, поэтому однажды он вполне может не заметить вызов геттера при работе с объектом Java-класса.<br/>
</li>
</ul><br/>
</div>
                    </div><br/>
<h2>4. Различные способы объявления функции</h2><br/>
И сразу вопрос. Какой текст будет выведен при запуске данного кода?<br/>
<br/>
<pre><code class="kotlin">fun example1() { System.out.println("example1") }
fun example2() = System.out.println("example2")
val example3   = { System.out.println("example3") }
fun example4() = { System.out.println("example4") }
val example5   = run{ System.out.println("example5") }

fun main()
{
    example1()
    example2()
    example3()
    example4()
    example5
}</code></pre><br/>
<div class="spoiler" role="button" tabindex="0">
                        <b class="spoiler_title">Ответ</b>
                        <div class="spoiler_text"> example5<br/>
 example1<br/>
 example2<br/>
 example3<br/>
</div>
                    </div><br/>
Как же так получилось?<br/>
<br/>
Ну, в случае с <code>example5</code> всё понятно — здесь используется так называемый <a href="https://kotlinlang.org/docs/scope-functions.html#function-selection" rel="nofollow noopener noreferrer"><u>non-extension run</u></a>, который сразу же выполняет требуемый блок кода. Так что этот пункт — просто «ловушка» (строка с <code>example5</code> внутри <code>main()</code> даже не обязательна, текст всё равно выведется при входе в <code>main()</code>, поскольку <code>val example5</code> стоит на top level). И вообще, так писать не надо :)<br/>
<br/>
<code>example1</code> и <code>example2</code> — это два эквивалентных способа объявления функции. Первый — классический Java-стиль, второй — это <a href="https://kotlinlang.org/docs/functions.html#single-expression-functions" rel="nofollow noopener noreferrer">синтаксический сахар</a> Kotlin. Результат для них, разумеется, тоже будет одинаков.<br/>
<br/>
<code>example3</code>, в отличие двух предыдущих, является лямбдой. Её вызов также приводит к ожидаемому результату.<br/>
<br/>
А вот почему вызов <code>example4</code> ничего не вывел? Дело в том, что <code>example4</code> — это функция, которая лишь возвращает лямбду, но не выполняет её. Чтобы было понятнее, её объявление эквивалентно следующей конструкции:<br/>
<br/>
<pre><code class="kotlin">val lambdaForExample4 = { System.out.println("example4") }
fun example4(): () -&gt; Unit
{
    return lambdaForExample4
}</code></pre><br/>
Поэтому в <code>main()</code> строка <code>example4()</code> просто возвращает нам эту самую <i>lambdaForExample4</i>, т.е. до <code>println</code> исполнение не доходит.<br/>
<br/>
Получается, что если мы действительно хотим увидеть строку «example4» в консоли, соответствующий вызов в <code>main()</code> нужно заменить на:<br/>
<br/>
<pre><code class="kotlin">example4()()</code></pre><br/>
Первые «круглые скобки» вернут нам лямбду, а вот вторые уже вызовут её и напечатают желанную строку.<br/>
<br/>
Таким образом, опасность состоит в том, что при написании классов на Kotlin часто приходится смешивать классический «Java-стиль» объявления методов (если метод содержит несколько выражений) с новым «Kotlin-стилем» (для методов из одного выражения). И есть риск вместо правильного варианта «example2» поставить по-привычке фигурные скобки… и получить «example4» — вызов которого, фактически, ничего не сделает, но компилятор при этом ругаться не будет. Такую ошибку сложно заметить в рантайме. Особых рекомендаций по предотвращению этой ошибки нет — нужно просто быть внимательным при записи single-expression functions.<br/>

И получается весьма неожиданный результат…\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\nСпособов избежать данной проблемы много. Можно явно блокировать поток через \u003Ccode\u003EThread.sleep()\u003C\u002Fcode\u003E, можно оборачивать в \u003Ccode\u003Esynchronized\u003C\u002Fcode\u003E лишь блок кода, а можно \u003Ca href=\"https:\u002F\u002Fkotlinlang.org\u002Fdocs\u002Fshared-mutable-state-and-concurrency.html#mutual-exclusion\" rel=\"nofollow noopener noreferrer\"\u003E\u003Cu\u003Eвоспользоваться мьютексами\u003C\u002Fu\u003E\u003C\u002Fa\u003E — в отличие от \u003Ccode\u003Esynchronized\u003C\u002Fcode\u003E, они корректно работают с \u003Ccode\u003Edelay()\u003C\u002Fcode\u003E в корутинах (и результат будет правильный, т.е. как в случае с \u003Ccode\u003EThread.sleep()\u003C\u002Fcode\u003E).\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Ch2\u003E3. Вызовы \u003Ci\u003Egetter\u003C\u002Fi\u003E'ов из Java-кода, выглядящих как поля класса\u003C\u002Fh2\u003E\u003Cbr\u002F\u003E\nОдна из фич Kotlin (являющаяся, как и большинство других, синтаксическим сахаром) — это \u003Ci\u003E\"\u003Cabbr title=\"по крайней мере, так их зовёт Android Studio\"\u003Esynthetic properties\u003C\u002Fabbr\u003E\"\u003C\u002Fi\u003E, то есть возможность обращения к уже существующим геттерам\u002Fсеттерам Java-класса так, как будто вы обращаетесь к полям этого класса. Иначе говоря, Kotlin (неявно) автоматически генерирует эти properties для всех подходящих Java-классов.\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Cdiv class=\"spoiler\" role=\"button\" tabindex=\"0\"\u003E\n                        \u003Cb class=\"spoiler_title\"\u003EПростой пример использования synthetic properties\u003C\u002Fb\u003E\n                        \u003Cdiv class=\"spoiler_text\"\u003EВозьмём стандартный Android-класс \u003Ci\u003Eandroid.view.View\u003C\u002Fi\u003E, в котором уже имеется поле \u003Ci\u003EmTag\u003C\u002Fi\u003E и соответствующие ему геттер\u002Fсеттер:\u003Cbr\u002F\u003E\n\u003Cpre\u003E\u003Ccode class=\"java\"\u003Epublic class View implements Drawable.Callback, KeyEvent.Callback\n{\n    protected Object mTag = null;\n    ...\n    public Object getTag()  { return mTag; }\n    public void setTag(final Object tag)  { mTag = tag; }\n}\u003C\u002Fcode\u003E\u003C\u002Fpre\u003E\u003Cbr\u002F\u003E\nТеперь, если при разработке на Kotlin мы захотим узнать значение \u003Ci\u003EmTag\u003C\u002Fi\u003E для какого-то объекта этого класса, то это можно будет сделать двумя способами:\u003Cbr\u002F\u003E\n\u003Cpre\u003E\u003Ccode class=\"kotlin\"\u003Eval myView: android.view.View = \u002F* какой-то инициализатор *\u002F\n...\nval myTag1 = myView.getTag()  \u002F\u002F старый способ с использованием геттера\nval myTag2 = myView.tag  \u002F\u002F новый способ с использованием synthetic properties \u003C\u002Fcode\u003E\u003C\u002Fpre\u003E\u003Cbr\u002F\u003E\nТаким образом, Kotlin даёт нам удобную возможность обращаться к \u003Ci\u003EmTag\u003C\u002Fi\u003E так, как будто мы обращаемся к полю с именем \u003Ci\u003Etag\u003C\u002Fi\u003E (это имя берётся из имён геттера\u002Fсеттера). Оба этих способа совершенно эквивалентны — но «старый» способ не нравится Android Studio, и она выдаст предупреждение при его использовании.\u003Cbr\u002F\u003E\n\u003C\u002Fdiv\u003E\n                    \u003C\u002Fdiv\u003E\u003Cbr\u002F\u003E\nКазалось бы, простая и удобная фича — что же в ней может быть опасного? А то, что геттер может быть не тривиальный (т.е. не вида \u003Ci\u003E«return value»\u003C\u002Fi\u003E), а довольно \u003Cb\u003Eсложный\u003C\u002Fb\u003E — то есть, выполняться продолжительное время. Что может привести к проблемам при использовании подобного кода (пример для класса \u003Ci\u003EFirebaseDatabase\u003C\u002Fi\u003E, входящего состав Firebase):\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Cpre\u003E\u003Ccode class=\"kotlin\"\u003Eval dbTableNames = arrayOf(\"first\", \"second\", \"third\", \"fourth\")\n...\n\nfun clearAllDbTables(db: FirebaseDatabase)\n{\n    for (currentName in dbTableNames)\n        db.reference.child(currentName).setValue(null)\n}\u003C\u002Fcode\u003E\u003C\u002Fpre\u003E\u003Cbr\u002F\u003E\nС первого взгляда, в этом коде не видно каких-либо проблем. Но на самом деле, поле \u003Ci\u003Ereference\u003C\u002Fi\u003E объекта \u003Ci\u003Edb\u003C\u002Fi\u003E — это не поле, а вызов геттера \u003Ci\u003EgetReference()\u003C\u002Fi\u003E класса \u003Ci\u003EFirebaseDatabase\u003C\u002Fi\u003E, который не просто возвращает нам уже существующий объект класса \u003Ci\u003EDatabaseReference\u003C\u002Fi\u003E, а каждый раз создаёт его заново.\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\nТо есть, в цикле \u003Ci\u003Efor\u003C\u002Fi\u003E поочерёдно будут созданы 4 одинаковых объекта, что плохо — хотя мы могли бы избежать этого, если бы просто вынесли получение \u003Ci\u003Ereference\u003C\u002Fi\u003E за пределы цикла:\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Cpre\u003E\u003Ccode class=\"kotlin\"\u003Efun clearAllDbTables(db: FirebaseDatabase)\n{\n    val rootReference = db.reference\n    \n    for (currentName in dbTableNames)\n        rootReference.child(currentName).setValue(null)\n}\u003C\u002Fcode\u003E\u003C\u002Fpre\u003E\u003Cbr\u002F\u003E\nВ общем, в первом варианте кода налицо проблема с производительностью из-за того, что геттер \u003Ci\u003EgetReference()\u003C\u002Fi\u003E не тривиальный. Способ избежать проблемы очевиден — нужно проверять, является ли геттер тривиальным, и избегать его множественного вызова в противном случае.\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\nНаверняка кто-то возразит — «постойте, ведь в других языках программирования properties тоже есть, и никто не жалуется!».\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Cdiv class=\"spoiler\" role=\"button\" tabindex=\"0\"\u003E\n                        \u003Cb class=\"spoiler_title\"\u003EДа, это так, но в случае с Java и Kotlin есть пара нюансов\u003C\u002Fb\u003E\n                        \u003Cdiv class=\"spoiler_text\"\u003E\u003Cul\u003E\n\u003Cli\u003Eво-первых, в Java нет properties, поэтому при кодинге на Kotlin привыкший к Java программист может просто не знать о том, что при этом производится вызов геттера;\u003C\u002Fli\u003E\n\u003Cli\u003Eво-вторых, Kotlin всячески подталкивает программиста к использованию полей вместо методов при разработке собственного класса. Например, если в класс нужно добавить поле, доступное сторонним классам только на чтение, то рекомендуется делать это как:\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Cpre\u003E\u003Ccode class=\"kotlin\"\u003E\nvar fieldName: FieldType\n    private set\u003C\u002Fcode\u003E\u003C\u002Fpre\u003E\u003Cbr\u002F\u003E\nвместо объявления private-поля \u003Ci\u003E«fieldName»\u003C\u002Fi\u003E с публичным тривиальным геттером. Таким образом, в Kotlin мы намного чаще обращаемся к полям класса напрямую (т.е. не через геттеры), чем в Java. И в итоге программист \u003Cb\u003Eпривыкает\u003C\u002Fb\u003E к этому, поэтому однажды он вполне может не заметить вызов геттера при работе с объектом Java-класса.\u003Cbr\u002F\u003E\n\u003C\u002Fli\u003E\n\u003C\u002Ful\u003E\u003Cbr\u002F\u003E\n\u003C\u002Fdiv\u003E\n                    \u003C\u002Fdiv\u003E\u003Cbr\u002F\u003E\n\u003Ch2\u003E4. Различные способы объявления функции\u003C\u002Fh2\u003E\u003Cbr\u002F\u003E\nИ сразу вопрос. Какой текст будет выведен при запуске данного кода?\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Cpre\u003E\u003Ccode class=\"kotlin\"\u003Efun example1() { System.out.println(\"example1\") }\nfun example2() = System.out.println(\"example2\")\nval example3   = { System.out.println(\"example3\") }\nfun example4() = { System.out.println(\"example4\") }\nval example5   = run{ System.out.println(\"example5\") }\n\nfun main()\n{\n    example1()\n    example2()\n    example3()\n    example4()\n    example5\n}\u003C\u002Fcode\u003E\u003C\u002Fpre\u003E\u003Cbr\u002F\u003E\n\u003Cdiv class=\"spoiler\" role=\"button\" tabindex=\"0\"\u003E\n                        \u003Cb class=\"spoiler_title\"\u003EОтвет\u003C\u002Fb\u003E\n                        \u003Cdiv class=\"spoiler_text\"\u003E example5\u003Cbr\u002F\u003E\n example1\u003Cbr\u002F\u003E\n example2\u003Cbr\u002F\u003E\n example3\u003Cbr\u002F\u003E\n\u003C\u002Fdiv\u003E\n                    \u003C\u002Fdiv\u003E\u003Cbr\u002F\u003E\nКак же так получилось?\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\nНу, в случае с \u003Ccode\u003Eexample5\u003C\u002Fcode\u003E всё понятно — здесь используется так называемый \u003Ca href=\"https:\u002F\u002Fkotlinlang.org\u002Fdocs\u002Fscope-functions.html#function-selection\" rel=\"nofollow noopener noreferrer\"\u003E\u003Cu\u003Enon-extension run\u003C\u002Fu\u003E\u003C\u002Fa\u003E, который сразу же выполняет требуемый блок кода. Первый — классический Java-стиль, второй — это \u003Ca href=\"https:\u002F\u002Fkotlinlang.org\u002Fdocs\u002Ffunctions.html#single-expression-functions\" rel=\"nofollow noopener noreferrer\"\u003Eсинтаксический сахар\u003C\u002Fa\u003E Kotlin. Её вызов также приводит к ожидаемому результату.\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\nА вот почему вызов \u003Ccode\u003Eexample4\u003C\u002Fcode\u003E ничего не вывел? Дело в том, что \u003Ccode\u003Eexample4\u003C\u002Fcode\u003E — это функция, которая лишь возвращает лямбду, но не выполняет её. И есть риск вместо правильного варианта «example2» поставить по-привычке фигурные скобки… и получить «example4» — вызов которого, фактически, ничего не сделает, но компилятор при этом ругаться не будет. Такую ошибку сложно заметить в рантайме. Особых рекомендаций по предотвращению этой ошибки нет — нужно просто быть внимательным при записи single-expression functions.\u003Cbr\u002F\u003E\n\u003Cbr\u002F\u003E\n\u003Cb\u003EP.S.\u003C\u002Fb\u003E Данная статья не ставит цель создать негативное впечатление о Котлине.
